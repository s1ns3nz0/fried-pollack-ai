# Red AKS + kagent + ArgoCD Deployment

This deployment keeps `fried-pollack-ai` as a deterministic, coarse-grained MCP
ToolServer. kagent can trigger `run_engagement` and summarize the JSON report,
but the Kubernetes surface is restricted by `ALLOWED_RANGE_MODES`.

> **Reviewers / judges:** this file documents the author's live deployment and
> hardcodes tenant-specific names. To stand the stack up in **your own Azure
> subscription** from a clean checkout, follow **[JUDGE-DEPLOY.md](./JUDGE-DEPLOY.md)**
> (Path B), which uses the `judge.bicepparam` / `judge-sim.bicepparam` templates
> and `judge.env.example` instead of the author-specific values below.

## Boundary

Red agent workloads must run in the red plane, not in the simulation target
cluster.

| Plane | Resource group | AKS | Purpose |
|---|---|---|---|
| red | `dah-red-rg` | `dah-red-aks` | kagent, MCP ToolServer, red ArgoCD |
| sim | `dah-sim-rg` | `dah-sim-aks` | UAV simulation target range |
| soc | `dah-soc-rg` | `dah-soc-aks` | defender telemetry and alerting |

`dah-sim-aks` is a target. Do not deploy `fried-pollack-ai`, kagent, or red
ArgoCD there. The red agent may only reach approved target service endpoints
through firewall/NSG allowlists and target-profile DNS names. It must not hold
`dah-sim-aks` or `dah-soc-aks` kubeconfig/RBAC credentials.

## Bootstrap

> **Azure IaC moved.** All bicep and the `deploy-red-with-sim.sh` provisioning
> script now live in the sibling **[pollak-infra](https://github.com/s1ns3nz0/pollak-infra)**
> repo (private). Steps 1–2 and 5 below run
> from a `../pollak-infra` checkout; the `bicep/*` paths are relative to that
> repo. This app repo keeps only the on-cluster workload bootstrap (step 3) and
> the kustomize overlays.

1. Provision sim, then red, with existing-sim skip semantics (from `../pollak-infra`).

   ```bash
   cd ../pollak-infra && scripts/deploy-red-with-sim.sh && cd -
   ```

   The script:

   - checks for `dah-sim-rg/dah-sim-aks`,
   - skips sim deployment when the sim AKS already exists, so SOC-owned sim
     deployments are not overwritten,
   - deploys `bicep/sim.bicep` only when sim AKS is absent,
   - passes `dah-sim-vnet` as `simVnetResourceId` into the red deployment when
     that VNet exists,
   - deploys red without sim peering if the sim VNet is absent.

2. Provision red Azure resources with Bicep only.

   ```bash
   az deployment sub what-if \
     --location koreacentral \
     --template-file bicep/main.bicep \
     --parameters bicep/params/lab.bicepparam

   az deployment sub create \
     --location koreacentral \
     --template-file bicep/main.bicep \
     --parameters bicep/params/lab.bicepparam
   ```

3. Bootstrap kagent and the red ToolServer.

   ```bash
   scripts/bootstrap-red-agent.sh
   ```

   The script (Azure infra assumed already provisioned from pollak-infra;
   `BOOTSTRAP_APPLY_BICEP=true` re-applies it against a `../pollak-infra` checkout):

   - imports the private ACR mirror image required by the bundled Postgres chart,
   - injects the recreated ToolServer managed identity client ID into the
     rendered Kubernetes ServiceAccount,
   - creates the kagent Azure OpenAI secret from the Azure OpenAI account key,
   - installs `kagent-crds` and `kagent` Helm charts,
   - applies the red overlay.

4. Create or select the role-separated red managed identities.

   ```bash
   az identity show \
     --resource-group dah-red-rg \
     --name dah-red-kagent-mi \
     --query clientId -o tsv

   az identity show \
     --resource-group dah-red-rg \
     --name dah-red-toolserver-mi \
     --query clientId -o tsv
   ```

5. Workload Identity and role assignments are handled by Bicep.

   ```bash
   az deployment sub create \
     --name red-plane-current \
     --location koreacentral \
     --template-file bicep/main.bicep \
     --parameters bicep/params/lab.bicepparam
   ```

6. Grant identities only their role-specific permissions.

   Bicep grants:

   - kubelet `AcrPull` on red ACR,
   - ToolServer identity `Storage Blob Data Contributor` on red storage,
   - kagent identity `Cognitive Services OpenAI User` on the Azure OpenAI account,
   - configured operators `Azure Kubernetes Service RBAC Cluster Admin` on red AKS.

7. Configure GitHub Actions:

   - Repository variables: `ACR_NAME`
   - Repository secrets: `AZURE_CLIENT_ID`, `AZURE_TENANT_ID`,
     `AZURE_SUBSCRIPTION_ID`

8. Apply the red ArgoCD root app after installing ArgoCD.

   ```bash
   kubectl apply -f deploy/argocd/root-app.yaml
   ```

## GitOps Flow

On `main`, CI runs tests and gates, builds `fried-pollack-ai:${GITHUB_SHA}`,
pushes it to the red ACR, then updates the red AKS overlay with the same
immutable tag. Red ArgoCD syncs the committed overlay into `dah-red-aks`.

## Scope Boundary

Initial red AKS deployment allows `container` and `sitl` range modes only.
`hil` and `live` remain disabled in deployment configuration.

Default result flow writes immutable run artifacts to red storage. SOC ingest is
optional and only enabled when `emit_soc=true` and a target profile provides an
approved SOC ingest endpoint.
